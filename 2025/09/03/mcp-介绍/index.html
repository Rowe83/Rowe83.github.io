什么是 MCP？

MCP（Model Context Protocol，模型上下文协议）是一套面向“AI 客户端 ↔ 外部工具/数据源”的开放协议与约定，用于让大语言模型在安全、可控且可审计的前提下，访问文件系统、数据库、API、知识库等外部能力。它把这些外部能力抽象为「可发现、可声明、可调用」的服务，使得不同模型与不同工具之间能够以统一的方式进行集成。

为什么需要 MCP？

  • 统一接入：摆脱各家“插件/工具”形态不一致的问题，用统一规范对接任意能力。
  • 安全可控：以最小权限、可审计日志、可声明能力的方式暴露外部资源，便于企业合规。
  • 复用与沉淀：工具以 Server 的形式复用在不同模型/应用中，避免重复集成成本。
  • 可观测：调用均通过协议层，便于记录、回放与问题定位。

核心概念

  • Client（客户端）：承载模型推理与交互的应用（如聊天应用、IDE 扩展、Agent 框架）。
  • Server（服务端）：对外声明一组可用的工具、资源与提示（prompts），并实现其逻辑。
  • Tools（工具）：可被调用的操作（例如读取文件、发送 HTTP 请求、查询数据库）。
  • Resources（资源）：可被引用/浏览的外部数据（例如目录、文档、知识库条目）。
  • Prompts（提示）：Server 暴露的可复用 Prompt 片段，供 Client 组合调用。

工作流程（简化）

  1. 发现能力：Client 连接到某个 MCP Server，读取其能力声明（tools、resources、prompts）。
  2. 计划调用：模型根据用户目标与可用能力，决定是否调用某个 tool 或访问 resource。
  3. 执行与反馈：调用结果返回给模型/客户端；若需后续操作，继续计划与调用。
  4. 记录与审计：整个过程可被记录，便于回放与安全审计。

最佳实践与安全建议

  • 最小权限：仅暴露必要目录/API，并限定只读或特定操作。
  • 明确边界：为工具声明清晰的输入/输出与副作用，便于审计与回放。
  • 隔离运行：将 Server 进程与业务服务隔离，必要时使用容器/沙箱。
  • 观测与日志：记录调用链路（时间、入参、结果、错误），建设可视化面板。
  • 版本管理：为工具与资源声明版本/能力变更，避免“隐式破坏”。

典型场景

  • 研发效能：读写项目文件、运行测试、调用 CI/CD API，辅助代码变更与评审。
  • 数据助理：联通数据库、数据仓库与 BI，支持自然语言查询与报表生成。
  • 知识问答：对接企业知识库、文档搜索与检索增强生成（RAG）。
  • 业务自动化：跨系统编排 API，实现需要人机协作的工作流。

参考与生态

  • 规范与 SDK：@modelcontextprotocol/* 相关仓库与文档
  • 现成 Server：filesystem、browser、git、slack、jira 等社区实现
  • 客户端支持：Claude Desktop、编辑器/Agent 框架的适配中

如果你正在评估“让模型安全地动用外部能力”的方式，MCP 提供了一个更标准、更工程化的起点。
